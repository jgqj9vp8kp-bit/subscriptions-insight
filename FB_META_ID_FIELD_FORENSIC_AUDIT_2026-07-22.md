# Forensic-расследование: какое поле является источником Meta Campaign ID

Дата: 2026-07-22. Режим: forensic investigation only — код не менялся, исправления не
предлагались, архитектура не проектировалась. Все цифры в этом документе получены
прямыми read-only SQL-запросами к продакшн Supabase Postgres (`supabase db query
--linked`) через таблицы `transactions` (33 638 строк с metadata) и
`capsuled_facebook_stats` (231 строка, уровень исключительно `campaign`). ClickHouse
(`fact_user_cohorts`) в этой сессии недоступен напрямую (см. память
`clickhouse-verification-workflow`), но `transactions.normalized_payload` — это тот же
JSON, который `transactionMapper.ts` построчно копирует в ClickHouse без
трансформации полей campaign/utm, поэтому выводы переносяты на warehouse без потери
достоверности. Всего проверено 231 реальный Meta Campaign ID (100% campaign-level
записей, которые есть в системе) против 33 638 транзакций.

## 0. Главный вывод (коротко)

**`campaign_id` = `utm_campaign` дословно, без изменений, в 100% случаев (0
исключений на всей популяции). Это поле остаётся лучшим и единственным разумным
источником Meta Campaign ID.** Ни `utm_term`, ни `utm_content` НИ РАЗУ не совпали ни с
одним из 231 реальных Meta Campaign ID (0.0% покрытия у обоих). `utm_id` не даёт ни
одного нового совпадения сверх того, что уже даёт `utm_campaign` — он строго
подмножество. Гипотеза «мы сравниваем не то поле» **не подтвердилась**. Расхождения
(55 из 231 ID, 23.8% по количеству, но только 4.9% по spend) объясняются дублированными
кампаниями («- Copy» в имени) и новыми рынками без конверсий — не выбором неверного
поля.

## 3. Проверка гипотезы "семейства Meta ID" (`utm_campaign→Campaign`, `utm_term→AdSet`, `utm_content→Ad`)

Данные **структурно поддерживают**, но **не могут окончательно подтвердить** эту
гипотезу — и вот почему, с попыткой её опровергнуть, как и просил запрос:

**За гипотезу:** в реальных строках campaign/content/term — это три разных, но близких
числовых ID одного порядка величины, отличающиеся на небольшую дельту в последних
цифрах (типичный паттерн Meta: campaign/adset/ad создаются в Ads Manager почти
одновременно и получают соседние ID). Пример (utm_source=22):
```
utm_campaign = 120250632304960028
utm_content  = 120250632304970028   (+10 000 к campaign)
utm_term     = 120250632304980028   (+20 000 к campaign)
utm_id       = 120250632304980028_v2_s08   (= utm_term + суффикс)
```
Это ровно то, что ожидалось бы, если content/term — реальные Meta object ID соседнего
уровня (adset/ad), а не мусор.

**Против гипотезы / попытка опровержения:** у нас **нет ни одной строки уровня
`adset`/`ad`** в `capsuled_facebook_stats` — таблица на 100% состоит из `level =
'campaign'` (231/231). Значит, "подтвердить", что конкретный `utm_term` реально равен
Meta AdSet ID, а `utm_content` — Meta Ad ID, **нельзя без ad-level экспорта из Meta**,
которого в системе просто нет. Единственное, что можно доказать данными — это то, что
`utm_term`/`utm_content` **никогда не являются валидным Campaign ID** (0/231 совпадений
против всех известных campaign_id, включая 55 "проблемных"). Это необходимое, но не
достаточное условие для гипотезы AdSet/Ad.

**Дополнительное опровержение по байеру**: гипотеза "utm_id = utm_term + суффикс" —
**не универсальна**. Для `utm_source = "4"` (байер Ivan) `utm_id` **точно равен**
`utm_campaign` (без суффикса, без term) в 100% случаев (3 382/3 382 non-null строк с
utm_source=4). Суффиксный паттерн `_v2_sNN` появляется **только** у части байеров
(`utm_source = "22"` — 2 859/2 859 строк, `"19"` — 394/398, `"ig"` — 191/1 668,
`"th"` — 1/7). У `utm_source = "tiktok"` `utm_id` — это нативный TikTok ad id (короче,
другой формат, например `1870680575507634`), никак не связанный с Meta campaign/adset/ad
ID вообще. **Вывод: это не единая архитектура полей, а разные шаблоны URL,
самостоятельно настроенные каждым байером/платформой** — то есть `utm_id` не имеет
единого стабильного значения во всей системе и не может быть источником Meta Campaign ID
как поле в общем случае.

## 5. Внутренние суффиксы `utm_id` (`_v1`, `_v2`, `_s01`…`_s11`)

Обнаруженные в проде варианты: `_v2_s01` … `_v2_s11` (наблюдался диапазон s01–s11, во
всех случаях префикс версии — `v2`, других версий в выборке не встречено).

**Вывод: это внутренний идентификатор FunnelFox/байера, не часть Meta ID.**
Доказательства:
1. Meta Campaign/AdSet/Ad ID — всегда чисто числовые (проверено на 231 реальном ID и
   на всех utm_campaign/utm_term/utm_content в проде — ни одного alphanumeric ID с
   подчёркиванием). Meta API физически не отдаёт ID с суффиксами вида `_v2_s08`.
2. Снятие суффикса (`utm_id_norm`) не добавило **ни одного** нового совпадения с
   реальными Meta Campaign ID сверх того, что уже даёт `utm_campaign` (см. §2) — то
   есть суффикс не "мешает" распознать валидный Meta ID, потому что под ним и без него
   лежит `utm_term` (другой Meta-объект), а не campaign_id.
3. Суффикс появляется только у части байеров/шаблонов (§3) — типичное поведение
   внутренней разметки "версия шаблона трекинга / номер сплита" (`v2` = вторая версия
   URL-шаблона, `sNN` = номер варианта/сплита в рамках этого шаблона), а не
   Meta-нативного идентификатора.

## 7. Найденные противоречия

1. **Два разных определения "campaign_id" в кодовой базе.** В пайплайне транзакций/когорт
   (`palmerTransform.ts` → `transactionMapper.ts`) `campaign_id` = `campaign_id ||
   utm_campaign` (utm_campaign приоритетен по факту, т.к. отдельной колонки нет). Но в
   пайплайне лидов (`funnelfoxLeadsTransform.ts:191`, `funnelfox-leads-sync/index.ts:122`)
   `campaign_id` = `campaign_id || utm_content || utm_term || adset_id || ad_id` —
   **utm_campaign в этом списке приоритетов вообще отсутствует**. Это прямое текстовое
   свидетельство того, что кто-то в проекте уже закладывал возможность, что "campaign_id"
   в URL FunnelFox может означать не `utm_campaign`, а `utm_content`/`utm_term`. На
   практике это не влияет на авторитетные когорты — таблица `funnelfox_leads` сейчас
   пуста по `campaign_id`/`origin` (0 из 157 строк), код не активен на реальных данных —
   но сама несогласованность реальна и стоит зафиксировать.
2. **`capsuled_facebook_stats` — не стабильный snapshot.** Сравнение с прошлым аудитом
   (22 confirmed pairs, 2026-07-19) показало нулевое пересечение с сегодняшними 9
   one-digit парами. Это означает, что любые "матрицы совпадений" типа этой имеют
   срок годности, привязанный к последней синхронизации FB, а не являются статическим
   фактом о данных.
3. **`utm_id` присутствует в 42.6% транзакций с metadata, но НЕ добавляет ни одного
   уникального совпадения** к тому, что уже даёт `utm_campaign` — притом что по смыслу
   поля (согласно самому неймингу "utm_id") можно было бы ожидать, что это наиболее
   "формальный" campaign/ad identifier. Данные это не подтверждают.

---

## 8. Случаи, где Meta Campaign ID лучше совпадает НЕ с utm_campaign

**Не найдено ни одного такого случая** на всей проверенной популяции (231 Meta Campaign
ID × 6 полей-кандидатов = 1 386 проверок совпадения). Ни `utm_term`, ни `utm_content` не
совпали точно ни разу (0/231 каждое). `utm_id` совпадает только там, где он уже
идентичен `utm_campaign` (байер "4"); там, где он расходится с `utm_campaign` (байер
"22"/"19"/"ig", суффиксный шаблон), он расходится и с реальным Meta Campaign ID тоже — то
есть не "заменяет" `utm_campaign` лучшим значением, а просто содержит другой,
Meta-валидный, но НЕ campaign-уровня идентификатор (структурно похожий на ad/adset ID).

## 9. Окончательный вывод

**`campaign_id` (= `utm_campaign` дословно) должен использоваться как источник Meta
Campaign ID — это уже так, и альтернатив с бо́льшим покрытием на реальных данных не
найдено.** Гипотеза "мы сравниваем не то поле" **опровергнута** на 231 реальном Meta
Campaign ID и 33 638 транзакциях: единственное поле с ненулевым и содержательным
покрытием Meta Campaign ID — это `campaign_id`/`utm_campaign` (76.2% кампаний, 95.1%
spend). `utm_term` и `utm_content` — структурно похожи на Meta object ID (числовые,
близкие по величине к campaign_id), но эмпирически **никогда** не совпадают с известным
Campaign ID; наиболее вероятная роль (Ad Set ID / Ad ID) **не может быть подтверждена
напрямую**, потому что в системе физически нет ad-level Facebook экспорта
(`capsuled_facebook_stats.level` = только `'campaign'`, 231/231) — это конкретный пробел
в данных, а не в анализе.

Оставшиеся 23.8% кампаний без совпадения (4.9% по деньгам) по всем найденным признакам
относятся к уже описанному в предыдущем аудите (`fb-campaign-id-mapping-audit`)
механизму дублирования кампаний («- Copy» в имени, ID-соседи одной "семьи") плюс новый
кластер JP-кампаний без конверсий — не к выбору поля.

### Что нужно, чтобы закрыть оставшийся пробел (не предлагаю решение, только называю
недостающие данные):
- Facebook Ads Insights **на уровне adset/ad** (не только campaign) — тогда можно
  напрямую проверить `utm_term`/`utm_content` против настоящих Meta AdSet ID / Ad ID,
  а не только исключить их как Campaign ID.
- Более длинная история `capsuled_facebook_stats` (сейчас — только последний sync-батч)
  — чтобы one-digit-пары можно было сопоставлять во времени, а не по снимку одного дня.
